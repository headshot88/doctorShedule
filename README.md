# doctorSchedule
Выбран подход хранения расписания (каждый день) в HL-блоке b_hl_doctor_schedule с полями : 

- UF_USER_ID	Привязка к пользователю	ID врача (обязательное, индекс)
- UF_DAY_OF_WEEK	Список (enum)	Дни недели: Пн, Вт, Ср, Чт, Пт, Сб, Вс (обязательное)
- UF_IS_WORKDAY	Да/Нет (bool)	Флаг рабочего дня (если false – выходной)
- UF_TIME_FROM	Строка (или целое)	Время начала работы в формате H:i (09:00)
- UF_TIME_TO	Строка (или целое)	Время окончания работы в формате H:i (18:00)

Такая архитектура имеет плюсы:
- Простоту создания и настройки через административный интерфейс или API.
- Наличие ORM (классы, наследующие DataManager) для удобной выборки и обновления данных.
- Возможность расширения (добавление новых полей, например, для перерывов или исключений) без изменения основной структуры.
- Гибкость при интеграции с пользователями (привязка к UF_USER_ID).
